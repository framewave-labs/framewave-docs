# A2팀 통합 PRD

## 0. 문서 목적

이 문서는 `scy`, `lsg` 개인 저장소에 흩어진 A2팀 기획 문서를 합쳐 개발·QA·발표 기준으로 바로 사용할 수 있게 정리한 통합 PRD입니다.

기존 문서 간 충돌은 아래 최신 기준으로 정리합니다.

| 항목 | 통합 기준 |
| --- | --- |
| 서비스명 | KX GenStudio |
| 이미지 생성 | GPT Image 2.0 기준 |
| 영상 생성 | fal.ai 기반 Kling 3.0 / Seedance 2.0 |
| 역프롬프트 | Claude Sonnet |
| DB·스토리지 | Supabase |
| 내부 배포 | Railway |
| F002 스토리보드 그리드 | B팀 범위, A2팀 MVP에서 제외 |
| A2팀 핵심 | 이미지 생성, 컷별 영상 생성, 미디어 라이브러리, 역프롬프트 도전 |

## 1. 제품 개요

### 1.1 한 줄 정의

**KX GenStudio는 방송 제작자가 텍스트 기획과 레퍼런스를 바탕으로 이미지·영상 초안을 만들고, 생성 결과를 라이브러리에 저장·관리하는 생성형 AI 통합 엔진입니다.**

### 1.2 문제 정의

방송 제작 현장에서는 기획 단계에서 장면 콘셉트, 캐릭터, 배경, 영상 방향성을 빠르게 시각화해야 합니다. 하지만 현재 워크플로우는 이미지 생성, 영상 생성, 프롬프트 보정, 결과물 저장이 서로 다른 서비스에 흩어져 있어 다음 문제가 발생합니다.

| 문제 | 설명 |
| --- | --- |
| 작업 흐름 분산 | 이미지 생성, 영상 생성, 저장·관리 도구가 분리되어 반복 작업이 늘어납니다. |
| 결과물 관리 어려움 | 생성한 이미지와 영상이 흩어져 다시 찾거나 비교하기 어렵습니다. |
| 프롬프트 진입 장벽 | 방송 제작자는 AI 프롬프트 작성에 익숙하지 않을 수 있습니다. |
| 수정 비용 증가 | 영상 전체를 다시 생성하면 시간과 API 비용이 크게 증가합니다. |

### 1.3 목표

KX GenStudio의 목표는 방송 제작자가 한 흐름 안에서 다음 작업을 완료할 수 있게 하는 것입니다.

```text
프롬프트 입력
→ 배경/캐릭터 시트 또는 키이미지 생성
→ 생성 결과 저장
→ 라이브러리 조회
→ 스토리보드 9컷/Shot 프롬프트 기반 컷별 영상 생성
→ 마음에 들지 않는 컷만 재생성
→ 결과물 저장·조회
```

2주차 중간점검은 이미지 생성과 저장 흐름의 완성을 Pass 기준으로 보고, 4주차 최종평가는 컷별 영상 생성·재생성까지 확장하는 것을 A등급 목표로 둡니다.

## 2. 대상 사용자

| 사용자 | 핵심 니즈 |
| --- | --- |
| 방송 PD | 기획 의도를 빠르게 이미지·영상 초안으로 확인하고 싶습니다. |
| AD / 제작 보조 | 여러 컷의 시각 자료를 빠르게 만들고 정리해야 합니다. |
| 영상 기획자 | 발표용 레퍼런스와 콘셉트 이미지를 반복 생성·관리해야 합니다. |

### 2.1 대표 페르소나

**기획 PD**

- 드라마·예능 프리 프로덕션 단계에서 장면 분위기를 빠르게 시각화해야 합니다.
- 레퍼런스 이미지의 색감·조명·구도를 비슷하게 재현하고 싶지만 프롬프트 작성이 어렵습니다.
- 생성 결과를 저장해 회의와 발표에 재사용하고 싶습니다.

**조연출 AD**

- 촬영 전 감독에게 여러 컷의 콘셉트 이미지를 빠르게 보여줘야 합니다.
- 영상 초안을 생성하더라도 전체를 다시 만들기보다 일부 컷만 수정하고 싶습니다.
- API 비용을 의식하면서도 데모 가능한 품질을 확보해야 합니다.

## 4. 단계별 성공 기준

### 4.1 2주차 Pass 기준

2주차 중간점검은 아래 흐름이 실제 화면에서 끊기지 않고 동작하면 Pass로 봅니다.

```text
프롬프트 입력
→ GPT Image 2.0 이미지 생성
→ 결과 화면 표시
→ Supabase 저장
→ 라이브러리 조회
```

| ID | 요구사항 | 완료 기준 |
| --- | --- | --- |
| M001 | 이미지 프롬프트 입력 | 사용자가 프롬프트를 입력할 수 있습니다. |
| M002 | 이미지 생성 API 호출 | `gpt-image-2` 기준 이미지 생성 요청이 성공합니다. |
| M003 | 결과 표시 | 생성 이미지가 결과 영역에 표시됩니다. |
| M004 | 저장 | 생성 이미지를 Supabase Storage와 DB에 저장합니다. |
| M005 | 라이브러리 조회 | 저장된 이미지가 라이브러리에서 보입니다. |
| M006 | 상태 UI | 로딩, 실패, 빈 상태가 사용자에게 표시됩니다. |

### 4.2 B등급 방어 기준

| 기능 | 기준 |
| --- | --- |
| F001 | 이미지 생성·저장·조회 안정화 |
| F003-B | 단일 Shot 영상 생성·상태 조회·저장 |
| F005 | 이미지·영상 통합 라이브러리 |
| 배포 | Railway 또는 최종 확정 플랫폼에서 데모 URL 접근 가능 |

### 4.3 A등급 목표 기준

| 기능 | 기준 |
| --- | --- |
| F003-A Shot 세션 | Shot1~9를 세션으로 관리합니다. |
| 개별 Shot 상태 | 각 Shot의 `pending`, `processing`, `completed`, `failed` 상태가 표시됩니다. |
| 개별 재생성 | 마음에 들지 않는 Shot만 다시 생성할 수 있습니다. |
| 결과 보존 | 재생성하지 않은 Shot 결과는 유지됩니다. |
| 라이브러리 | 이미지와 영상 결과를 저장·조회합니다. |

### 4.4 S등급 도전 기준

| 단계 | 기준 |
| --- | --- |
| 1단계 추출 | 레퍼런스 이미지 업로드 후 Claude Sonnet으로 스타일·조명·구도·렌즈·색감·분위기를 분석합니다. |
| 2단계 수정 | 추출된 프롬프트를 사용자가 직접 수정할 수 있습니다. |
| 3단계 재생성 | 수정된 프롬프트로 GPT Image 2.0 이미지를 재생성합니다. |
| 4단계 저장 | 재생성 결과를 라이브러리에 저장합니다. |

## 5. 핵심 사용자 시나리오

### 5.1 이미지 생성과 저장

```text
1. 사용자가 이미지 생성 화면에 진입합니다.
2. 방송 장면 콘셉트 프롬프트를 입력합니다.
3. 비율과 품질 옵션을 선택합니다.
4. [생성하기] 버튼을 클릭합니다.
5. 로딩 상태가 표시됩니다.
6. GPT Image 2.0 생성 결과가 화면에 표시됩니다.
7. 결과물이 자동 또는 수동으로 라이브러리에 저장됩니다.
8. 라이브러리에서 생성물을 다시 확인합니다.
```

### 5.2 단일 Shot 영상 생성

```text
1. 사용자가 영상 생성 화면에 진입합니다.
2. B팀에서 받은 Shot 프롬프트와 9분할 그리드 이미지를 입력합니다.
3. 모델은 개발 테스트 시 Kling 3.0, 데모 시 Seedance 2.0을 선택합니다.
4. 영상 길이는 테스트 5초, 최종 데모 10초를 기본으로 둡니다.
5. 생성 요청 후 `request_id`를 받습니다.
6. FE는 5초 간격으로 상태를 조회합니다.
7. 완료되면 영상 플레이어에 결과를 표시하고 라이브러리에 저장합니다.
```

### 5.3 Shot 세션 기반 컷별 재생성

```text
1. 사용자가 Shot1~9 프롬프트와 그리드 이미지를 업로드합니다.
2. 세션이 생성되고 Shot 카드 9개가 표시됩니다.
3. [전체 생성]으로 Shot별 영상 생성을 순차 실행합니다.
4. 각 Shot 카드에서 상태와 결과 영상을 확인합니다.
5. 마음에 들지 않는 Shot만 프롬프트를 수정합니다.
6. [재생성] 버튼으로 해당 Shot만 다시 생성합니다.
7. 마음에 드는 Shot은 유지하고 최종 후보를 라이브러리에 저장합니다.
```

### 5.4 역프롬프트

```text
1. 사용자가 레퍼런스 이미지를 업로드합니다.
2. [프롬프트 추출] 버튼을 클릭합니다.
3. Claude Sonnet이 이미지의 스타일·조명·구도·렌즈·색감·분위기를 분석합니다.
4. 재현용 영문 프롬프트를 반환합니다.
5. 사용자가 프롬프트를 수정합니다.
6. [이 프롬프트로 이미지 생성] 버튼으로 GPT Image 2.0 이미지를 생성합니다.
7. 결과를 라이브러리에 저장합니다.
```

## 7. 데이터 모델

### 7.1 `media_items`

이미지와 영상을 단일 테이블로 관리합니다.

```sql
CREATE TABLE media_items (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  type TEXT NOT NULL CHECK (type IN ('image', 'video')),
  status TEXT NOT NULL DEFAULT 'completed'
    CHECK (status IN ('pending', 'processing', 'completed', 'failed')),
  prompt TEXT,
  model TEXT,
  aspect_ratio TEXT,
  duration INTEGER,
  storage_path TEXT,
  public_url TEXT,
  thumbnail_url TEXT,
  file_size INTEGER,
  fal_request_id TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_media_items_created_at ON media_items(created_at DESC);
CREATE INDEX idx_media_items_type ON media_items(type);
```

### 7.2 `storyboard_sessions`

Shot1~9를 하나의 작업 단위로 묶습니다.

```sql
CREATE TABLE storyboard_sessions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  grid_image_url TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'in_progress'
    CHECK (status IN ('in_progress', 'completed')),
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);
```

### 7.3 `storyboard_shots`

세션 내 개별 Shot 상태와 결과를 관리합니다.

```sql
CREATE TABLE storyboard_shots (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  session_id UUID NOT NULL REFERENCES storyboard_sessions(id) ON DELETE CASCADE,
  shot_number INTEGER NOT NULL CHECK (shot_number BETWEEN 1 AND 9),
  prompt TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'pending'
    CHECK (status IN ('pending', 'processing', 'completed', 'failed')),
  video_url TEXT,
  fal_request_id TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE (session_id, shot_number)
);

CREATE INDEX idx_storyboard_shots_session ON storyboard_shots(session_id, shot_number);
```

| `INVALID_INPUT` | 빈 프롬프트, 잘못된 파일 형식 | 입력값을 확인해 주세요. |
| `GENERATION_FAILED` | 이미지·영상 생성 실패 | 생성에 실패했습니다. 다시 시도해 주세요. |
| `API_LIMIT_EXCEEDED` | API 사용량 또는 비용 한도 초과 | 사용량 한도에 도달했습니다. PM에게 문의해 주세요. |
| `TIMEOUT` | 영상 생성 10분 초과 | 영상 생성 시간이 초과되었습니다. 다시 시도해 주세요. |
| `STORAGE_FAILED` | Supabase 저장 실패 | 결과 저장에 실패했습니다. 다시 시도해 주세요. |

## 13. QA 기준

### 13.1 이미지 생성

- 빈 프롬프트 제출 시 생성 요청이 발생하지 않습니다.
- 생성 중 버튼이 비활성화되고 로딩 상태가 표시됩니다.
- 생성 완료 후 결과 이미지가 표시됩니다.
- 저장 후 라이브러리에서 조회됩니다.
- API 실패 시 사용자 메시지가 표시됩니다.

### 13.2 영상 생성

- 요청 직후 `processing` 상태가 표시됩니다.
- 5초 간격 polling이 동작합니다.
- 완료 후 영상 플레이어가 표시됩니다.
- 10분 초과 시 타임아웃 처리됩니다.
- 실패 시 재시도 버튼이 표시됩니다.

### 13.3 Shot 세션

- Shot1~9가 순서대로 표시됩니다.
- 각 Shot의 상태가 독립적으로 표시됩니다.
- 특정 Shot만 재생성할 수 있습니다.
- 재생성하지 않은 Shot 결과가 유지됩니다.
- 새로고침 후에도 세션 상태를 복원할 수 있습니다.

### 13.4 라이브러리

- 이미지와 영상이 최신순으로 표시됩니다.
- 타입 필터가 동작합니다.
- 상세 모달에서 프롬프트, 모델, 생성일을 확인할 수 있습니다.
- 다운로드가 동작합니다.
- 데이터가 없을 때 빈 상태 UI가 표시됩니다.
